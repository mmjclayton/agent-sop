Severity: MEDIUM

## Findings

- **scripts/validate-state-transitions.sh:202-203** — Threshold comparison uses committed-diff size (`git diff --numstat "$base" -- 2>/dev/null`) which measures base-to-working-tree line changes. However, `commit_pnums` is collected at line 199 only from committed git log messages (`base..HEAD`). An agent with large uncommitted changes will see `loc` exceed the threshold but `commit_pnums` will be empty (no committed messages to grep). The check fires with "commit messages reference no P-numbers" and hard-blocks — even if the agent has done zero committed work yet. This is a false-positive risk for agents who draft large working-tree changes before committing. Fix: either measure `loc`/`files` from committed diff only (`git diff --numstat "${base}..HEAD"`) for consistency with `commit_pnums`, or populate `commit_pnums` from staged/unstaged diff comments when working-tree changes exist.

- **scripts/validate-state-transitions.sh:228** — Threshold gate uses AND (`&&`): session must be BELOW BOTH `threshold_loc` AND `threshold_files` to skip. A session with 200 LOC across 2 files (above loc threshold, below files threshold) is NOT skipped and proceeds to the intersection check. P44's equivalent condition is expressed as OR (either threshold exceeded → run the reviewer gate). This inconsistency means drift detection is harder to bypass on the loc axis than the files axis, but more importantly it means a tiny session with 100 LOC in 1 file (loc over threshold, files under) will still be drift-checked. Whether the AND logic is intentional is unclear from the comments. If threshold means "small enough to skip", OR is the more conservative choice. The docs (`update-sop.md` Step 3d: "Thresholds reuse the P44 config fields — the same 'too small to worry about' boundary applies") imply the same semantics as P44, which uses OR. Fix: change line 228 to `if [ "$loc" -lt "$threshold_loc" ] || [ "$files" -lt "$threshold_files" ]; then` if OR semantics are intended, or document explicitly that AND is a deliberate tightening.

- **scripts/validate-state-transitions.sh:208-209** — After fixture-mode (`--drift-loc`/`--drift-files` flags) is used to supply commit P-numbers, the same flag values are unconditionally re-read into `threshold_loc`/`threshold_files` at lines 208-209 (`threshold_loc="${DRIFT_LOC_OVERRIDE:-}"`). This means `--drift-loc 500` both sets `loc=500` (the session size) AND `threshold_loc=500` (the threshold). In `run-tests.sh` the harness passes `--drift-loc 500 --drift-files 10`, so `loc=500` and `threshold_loc=500`, meaning the gate condition `500 -lt 500` is false and the check proceeds — which is correct by coincidence. But if a fixture needed to test the below-threshold skip path with an explicit LOC, there is no way to independently set session size vs threshold. This is a fixture/test design limitation, not a production bug, but it means the under-threshold fixture case is untested. Recommendation: add a `--drift-loc-session` / `--drift-threshold-loc` pair, or document that the two roles are conflated by design.

- **scripts/validate-state-transitions.sh:237** — Scope-change regex `'^##+[[:space:]]+scope change\b'` uses `-i` (case-insensitive) via `grep -qiE`. The `-i` flag is present so "SCOPE CHANGE" and "Scope-Change" would match if the literal text matched, but "Scope-Change" with a hyphen would NOT match because the regex requires a space (`[[:space:]]`) between "scope" and "change". The SOP documentation in `update-sop.md` describes the escape hatch as `## Scope Change` (space, sentence case) and the fixture uses that exact form. The hyphen variant failing silently is acceptable IF the required form is documented clearly. Current wording in `update-sop.md` is "add a `## Scope Change` block" — specific enough. No action required, but worth noting in docs that "Scope-Change" (hyphenated) is not accepted.

- **scripts/validate-state-transitions.sh:152** — Project-hash normalization `sed 's|[^a-zA-Z0-9-]|-|g' | sed 's|^-||'` strips any leading hyphen after replacement, but a path like `/Users/matt_clayton/Projects/agent-sop` produces `Users-matt-clayton-Projects-agent-sop` (leading slash → leading hyphen → stripped). The memory dir is then constructed as `$HOME/.claude/projects/-Users-matt-clayton-Projects-agent-sop/...`. The leading `-` in the directory prefix is hardcoded at line 153 (`-$project_hash`), so the final path is `~/.claude/projects/-Users-matt-clayton-Projects-agent-sop/memory/...`. This matches the actual structure visible in the system context (`/Users/matt_clayton/.claude/projects/-Users-matt-clayton/memory/MEMORY.md`). The fix for underscores (non-alphanumeric → hyphen) is correct; no further issue here except that this is unverified for paths containing consecutive non-alphanumeric characters (e.g. `~/My__Projects/`) which would collapse to `My--Projects` — Claude Code's actual behaviour in that case is unknown. Low risk in practice.

- **.claude/commands/restart-sop.md (Step 0d inline script)** — The print statement at line 20 of the new block is missing a space before the parenthetical: `echo "In-flight declared: $pnums(from $resume)"`. This produces output like `In-flight declared: P46 P47 (from ...)` with no space before `(from`. Cosmetic only, but the reassertion message is what the agent reads at session start. Fix: `echo "In-flight declared: $pnums (from $resume)"`.

- **docs/benchmark/drift-fixtures/** — Only 3 fixture cases are present: legal-matching, illegal-mismatch, legal-scope-change. Missing: (a) the first-session / no-resume-file case (verifying clean skip), and (b) the below-threshold case (verifying the gate does not fire for small sessions). The harness hardwires `--drift-loc 500 --drift-files 10` to force over-threshold in every run, so there is no fixture-driven test of the threshold-skip path. The threshold skip code at line 228 is exercised only by inference. Recommendation: add `legal-below-threshold` and `legal-no-resume` fixture cases, or add a dedicated harness path for the no-resume case.

## Notes

- **Multi-agent correctness (question 7):** Each agent reads only its own `project_resume_<agent-id>.md`. If agent A declares P100 and agent B commits P100 without updating A's resume, A will not be falsely flagged — A's resume still lists P100 and A has no commits (in A's worktree) for B's work. The per-worktree commit range means sibling agent false-positives are structurally prevented. Correct behaviour.

- **Step ordering (question 6):** Step 3d follows Step 3c (state-transition validator). This ordering is sound: the transition validator checks the Backlog file is internally consistent BEFORE drift detection asks whether the work committed this session matches the declared item. No circular dependency with P44 Step 1b — Step 1b is a pre-commit gate on a different axis (review artifact existence), not on P-number alignment.

- **Threshold=0 edge case (question 3):** If `review_loc_threshold` is set to 0 in config, `threshold_loc=0`. Then `loc -lt 0` is always false (loc is non-negative), so every session exceeding 0 files would be drift-checked. This is user-intended if they set 0 (opt-in to always check). Not a bug, but worth a comment in the config template documenting that 0 means "always enforce".

- **P46 AC numbering discrepancy:** The Backlog item references "Step 2d" in the original ACs but the implementation landed at Step 3d. The ACs were updated inline to note the renumbering. No functional issue.

## Resolution (appended 2026-04-19, same session)

6 of 7 findings addressed. 1 re-assessed and kept as-is (explained below).

**MEDIUM 1 (uncommitted-work false-positive):** fixed. `commit_pnums` and `loc`/`files` now both measure the committed range `base..HEAD` exclusively. An added early-exit: when there are no committed commits yet (`base == HEAD_sha`), the check skips with "no session commits yet — skipping until after first commit."

**MEDIUM 2 (AND vs OR threshold):** re-assessed, NOT changed. The reviewer's claim that AND-skip is stricter than OR-fire is incorrect — by De Morgan, `skip iff NOT(fire)`, so `AND-skip` ≡ `OR-fire`. A 200-LOC single-file change correctly fires the drift check (loc over threshold, so the OR is true, so NOT(OR) is false, so don't skip). Comment expanded to prevent future confusion.

**MEDIUM 3 (flag collision):** fixed. Split `--drift-loc` into four distinct flags: `--drift-session-loc` / `--drift-session-files` (fixture-mode session size) and `--drift-threshold-loc` / `--drift-threshold-files` (threshold override). Fixture harness updated. A new `legal-below-threshold` fixture (with `.session-size` sidecar of `10 1`) exercises the below-threshold skip path; previously it was untested.

**LOW 4 (missing space in reassertion):** fixed. Added explicit space before `(from ...)` and trailing-whitespace trim on the P-numbers string.

**LOW 5 (fixture gaps):** added `legal-below-threshold` (session size under both thresholds) and `legal-no-declaration` (resume file has no P-numbers, check skips). Found and fixed another pipefail+errexit bug during this: grep on no-match returned 1, pipefail killed the script silently. Wrapped with `|| true`.

**LOW 6 (Scope-Change hyphen):** fixed. Regex now `scope[[:space:]-]+change` — accepts "Scope Change", "Scope-Change", "SCOPE CHANGE" case-insensitively.

**LOW 7 (consecutive hyphens in project-hash):** fixed. Added `sed 's|--*|-|g'` collapse in both the validator and the restart-sop reassertion snippet.
