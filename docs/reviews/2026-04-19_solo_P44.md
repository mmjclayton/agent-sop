## Findings

- `.claude/commands/update-sop.md:109` — `git diff --shortstat "$base" --` with a naked `--` terminator passes no pathspec, so the diff counts every changed file, not just tracked ones (e.g. untracked staged files are missed and `--` with no path is a no-op here). The awk field extraction `{print $4+$6}` is fragile: `--shortstat` output is locale-sensitive ("1 file changed, 3 insertions(+), 1 deletion(-)") and field positions shift when the deletion column is absent (net-add-only commit). A file with only additions produces "1 file changed, 3 insertions(+)" — `$6` is empty, `$4+$6` still works, but a deletion-only commit produces "1 file changed, 1 deletion(-)" where insertions are absent and `$4` is the deletion count, not insertions. This makes the LOC count wrong for deletion-only sessions. Suggested fix: use `git diff --shortstat "$base" | grep -oE '[0-9]+ insertion' | grep -oE '[0-9]+'` and `git diff --shortstat "$base" | grep -oE '[0-9]+ deletion' | grep -oE '[0-9]+'` separately, then sum; or use `git diff --numstat` and sum columns 1+2 with awk.

- `.claude/commands/update-sop.md:131` — Security-trigger list includes `sign` and `verify` as bare substring matches. These will false-positive on nearly any diff touching `validate-state-transitions.sh` (which contains `git rev-parse --verify`) and any docs file with "verify" in prose. A false positive routes an SOP-doc-only diff through `security-reviewer` instead of `code-reviewer`. The strings are too short and generic for substring matching without word-boundary anchors. Suggested fix: add word-boundary anchors (`\bsign\b`, `\bverify\b`) or replace with more specific strings (`signing`, `signature`, `verify_token`, `verify_password`).

- `.claude/commands/update-sop.md:138` — Step 1b item 6 states "the state-transition validator's Batch Log check (Step 3c) keeps this honest" (re: Batch Log must reference review path). The validator (`scripts/validate-state-transitions.sh` lines 253-258) only checks that a P-number appears anywhere in `docs/build-plans/phase-*.md` — it does not grep for `docs/reviews/`. The claim is false: an agent can write a Batch Log entry that mentions `P44` without naming the review artifact, and Step 3c will not block it. The gate has a hole — the review path is required by prose instruction but not by any automated check.

- `setup.sh` (no line) — `setup.sh` copies `review-template.md` to `docs/templates/review-template.md` but never creates `docs/reviews/` (the directory where review artifacts are written). On a fresh install, Step 1b will invoke the reviewer agent, which will fail when it tries to write `docs/reviews/YYYY-MM-DD_<agent-id>_P<n>.md` because the parent directory does not exist. The pre-migration skip ("if `docs/reviews/` is absent, skip with non-blocking warning") in Step 1b only triggers if the directory is absent — so on a fresh install that has never run `/update-agent-sop`, the warning fires correctly; but `setup.sh` is the install path and should create the directory proactively. Suggested fix: add `mkdir -p "$TARGET/docs/reviews"` to `setup.sh` alongside the other `mkdir -p` calls.

- `docs/sop/compliance-checklist.md:263` — R1's "best-effort retrospective" relies on `git show --stat` against the Batch Log commit range to measure session diff retrospectively, but `git show --stat` shows per-commit file counts, not the aggregate across the session range. A reviewer checking R1 for a multi-commit session will under-count LOC (each commit's stat is smaller than the session total). The check may incorrectly classify a large session as under-threshold and skip the artifact verification. Suggested fix: document that `git diff <merge-base>..<tip> --shortstat` is the correct retrospective measurement, not `git show --stat`.
