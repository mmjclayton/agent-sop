### Findings

1. **Rollup conflict is expected and acceptable.** The CLAUDE.md `<!-- recent-work-rollup:start -->` / `<!-- recent-work-rollup:end -->` section gets regenerated by each agent's `/update-sop` Step 8b from its own view of `docs/recent-work/`. When merges converge, git sees two different line additions inside the same sentinel block — textual conflict. The fix is mechanical and documented in the guide: pick either side, re-run the refresh, commit. The refresh is deterministic on identical directory contents so the post-merge state always converges.

2. **`refresh_recent_work_rollup` bash snippet has a zsh portability bug.** Two of three subagents (independent observations) noted that running the canonical snippet via macOS's default zsh leaks `local var=...` assignment lines into stdout, polluting `CLAUDE.md`. Workaround: invoke via `bash -c '...'`. **Follow-up to fix upstream:** rewrite the snippet in `/update-sop` Step 8b, `/migrate-to-multi-agent` Step 9, and `docs/guides/multi-agent-parallel-sessions.md` Section 3 to either (a) avoid `local` inside the compound output group, or (b) ship as `scripts/refresh-rollup.sh` with a `#!/usr/bin/env bash` shebang, or (c) explicitly prefix every documented invocation with `bash -c`.

3. **Agent-id hash collisions considered theoretical — not observed.** Three agent-ids computed as `sha256(worktree-path)[:6]`: `009b16`, `d56e9e`, `8a0be0`. All distinct. No clashes on any other filename surface.

4. **Spec drift caught by agent B is a legitimate improvement.** Backlog said START DATE is at cell M1; the real HST workbooks have the label at M2 and the serial at N2. Agent B implemented label-scanning (robust for real layout, spec-fallback for strict fixtures, null for legacy). Not a dogfood-mechanics finding but a product-quality one — the dogfood's task selection accidentally exposed latent spec accuracy, which is a useful side-effect.

5. **Migration script partial-miss on undated prose gotchas.** hst-tracker's `## Gotchas and Lessons` section contained 35+ bullets in the form `- **Topic**: body` with no leading date. The migration script's bullet parser only recognises `- YYYY-MM-DD:` and extracted 0 gotchas. Section kept intact as legacy; entries extract-able later once dates are assigned. **Follow-up to consider upstream:** extend the migration parser to recognise dateless bullets and assign a synthetic date (e.g. the file's earliest git-blame date for that section), or emit a warning instead of silent zero.
